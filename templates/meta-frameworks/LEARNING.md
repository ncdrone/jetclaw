# Learning — Meta Framework

*Be curious — always keep learning. Skills and knowledge are the engine of relevance and excellence.*
*v0.1*

## Purpose

This document defines how we acquire knowledge, develop competence, and grow capability over time. Not a curriculum — a methodology for learning anything systematically.

## Core Principle

**Learning is an active discipline, not passive accumulation.** Reading about a topic is not learning it. Building something real and validating its output against real data — that's learning. Knowledge without application is trivia.

## The Learning Lifecycle

### 1. Identify the Gap
- What do we need to know that we don't?
- Is this gap blocking progress? (Priority) Or building future capability? (Investment)
- How deep do we need to go? (Awareness vs. competence vs. expertise)

### 2. Research
- Use the research skill for structured investigation
- Multiple sources, adversarial thinking (what if the first source is wrong?)
- Time-box: set a deadline. Research without a cutoff expands infinitely.
- Output: a synthesis document, not a bookmark folder

### 3. Build
- Apply the knowledge to something real. A prototype, a script, a process.
- This is where understanding gets tested. If you can't build it, you don't understand it.
- Errors during building are the most valuable learning moments — capture them.

### 4. Validate
- Does what we built actually work? Against real data, real conditions, real users.
- External validation > self-assessment. Can we explain it to someone else? Does it match known benchmarks?
- If validation fails, the learning isn't complete — go back to step 2 with better questions.

### 5. Codify
- Turn the knowledge into a reusable artifact: a skill, a process, a reference document.
- The artifact should enable future-us (or a sub-agent) to apply this knowledge without re-learning it.
- This is how knowledge compounds instead of evaporating between sessions.

## Depth Levels

Not everything needs expertise. Match the investment to the need:

| Level | What It Means | When to Use |
|-------|---------------|-------------|
| **Awareness** | Know it exists, know where to learn more | Adjacent domains, context for conversations |
| **Competence** | Can do it with reference material | Most operational tasks, skill execution |
| **Expertise** | Can do it from memory, can teach it, can innovate | Core domains, competitive advantage |

## Learning Principles

**Learn in public.** Document as you go, not after. The daily memory files, skill docs, and process files are the learning trail. Future sessions benefit from today's notes.

**Teach to learn.** Writing a skill document forces you to organize knowledge. If you can't explain it clearly in a SKILL.md, you don't understand it well enough.

**Mistakes are data.** Every error, every wrong assumption, every failed experiment is information. Capture it. The lesson is worth more than the success.

**Cross-pollinate.** Knowledge from one domain informs another. Security thinking informs architecture. Outreach tactics inform content strategy. Don't silo knowledge — connect it.

**Spaced repetition through use.** Knowledge sticks when you use it regularly. Skills that aren't exercised decay. Periodic tasks and heartbeats can serve as natural repetition — keeping knowledge fresh.

## The Skill-Learning Cron

A scheduled learning task exists to systematically close knowledge gaps. Its behavior should be:

1. Check the knowledge domains table for gaps
2. Pick the highest-priority gap (aligned with current strategic constraint)
3. Spend the session researching or building toward closing it
4. Update the domain table and any relevant skill documents
5. Output: what was learned, what changed, what's next

Not random exploration. Directed learning toward identified gaps.

## Anti-Patterns

**Research addiction** — researching instead of building. Research is a means, not an end. If you've read 10 articles and built nothing, you're procrastinating.

**Expertise hoarding** — learning something but not codifying it. If the knowledge lives only in memory (which resets every session), it's lost.

**Depth without need** — spending days going deep on something that only needs awareness. Match investment to the actual requirement.

**Learning without validation** — assuming you understand because you read about it. Build something. Test it. That's when real understanding emerges.
